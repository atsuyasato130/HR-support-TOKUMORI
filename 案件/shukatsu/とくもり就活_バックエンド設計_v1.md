# とくもり就活 ― 本番バックエンド設計 v1（プロトタイプ→本番）

> 北極星（`とくもり就活_新メディア北極星_v1.md`）の本体＝二面ネットワークを、localStorageプロトタイプから本番へ。
> 本書は設計の正本。実装着手時は EnterPlanMode で本書を入力にして段階的に進める（認証/DB/課金/法務が絡むため）。
> 最終更新：2026-06-19。

## 0. Context
現状：会員プロフィール（`lib/profile-store.ts`=localStorage）、スカウト（`lib/scout-store.ts`=localStorage）、マッチング（`lib/matching.ts`=ヒューリスティック）、企業/学生プール（ダミー）。記事は公開層（microCMS想定/現ダミー）。
本番化で“実在”にするもの：**認証・会員DB・スカウトの永続化と通知・課金・AIマッチングの実推論・選考データのフライホイール**。コンプラ（§7）はRLSと同意設計で担保。

## 1. 技術スタック（推奨・実装時に再確認）
| 領域 | 採用 | 理由 |
|---|---|---|
| DB/認証/ストレージ | **Supabase**（Postgres＋Auth＋Storage＋RLS） | 認証・DB・行レベル権限が一体。RLSで“同意した分だけ企業に見せる”を強制できる |
| 認証方式 | Supabase Auth（**Magic Link＋Google／LINE OAuth**） | 学生はLINE/Google、企業はメール。パスワードレス |
| サーバ処理 | Next.js Route Handlers / Server Actions | 既存App Routerに同居 |
| 課金 | **Stripe**（Subscription＋従量＋請求書） | 掲載/スカウト/成功報酬を表現 |
| 通知 | メール（Resend等）＋アプリ内通知 | スカウト到着・返信 |
| AI | Anthropic API（マッチング/内定可能性/ES添削） | 既存`matching.ts`を実推論へ。参考値・人間検証(YMYL) |
| シークレット | `.env.local`（同期外・コミット禁止） | [[feedback_no_secrets_in_shared]] |

## 2. ロールと認証
- ロール：**student / recruiter(company) / agent / admin**（`profiles.role`）。
- 学生＝LINE/Googleで30秒登録（現`/signup`の本実装）。企業＝メール招待＋企業に紐付け。
- セッションは Supabase Auth。SSRは`@supabase/ssr`でcookieセッション。
- `/mypage`(student)・`/recruiter`(recruiter) はログイン必須・ロールガード。公開層は従来どおり匿名OK＋noindexは会員/管理のみ。

## 3. DBスキーマ（主要テーブル）
```
profiles            id(uuid,authユーザー) / role / name / university / grad_year / created_at
student_profiles    user_id(fk) / gakuchika(text[]) / certifications(text[]) / industries(text[])
                    / visibility_profile(bool) / visibility_records(bool)   ← 同意フラグ
selection_records   id / user_id(fk) / company_name / phase / result        ← ABABAの素
companies           id / name / industry / logo_url / summary / ai_note / plan_id / owner_user_id
company_stats       company_id / offer_rate / pass_rates(jsonb)             ← 匿名統計のみ
hr_voices           company_id / role / text / approved(bool)               ← 企業許諾
scouts              id / from_type(company|agent) / from_id / to_user_id / message / matched_on
                    / status(新着|既読|返信済み) / created_at
events / deadlines  （公開層・掲載データ）
plans / subscriptions / scout_usage                                          ← 課金
audit_log           actor / action / target / at                            ← コンプラ
citation_metrics    engine / query / cited / article_slug / measured_at      ← 学習ループ(§11)
```
記事/DATA LABは引き続きCMS（microCMS）。企業詳細の統計は`company_stats`（匿名）。

## 4. RLS（行レベル権限）＝コンプラの心臓
- `student_profiles` / `selection_records`：本人は全件。**企業/エージェントは `visibility_profile=true`（かつ進捗は `visibility_records=true`）の行のみ SELECT 可**。個人特定列はビュー経由で匿名化（イニシャル等）。
- `scouts`：`to_user_id=自分`(学生) or `from_id=自社`(企業) のみ。
- `companies`/`hr_voices`：公開は `approved=true` のみ。編集は `owner_user_id`。
- すべての公開系READは「同意・承認フラグ」をRLSで強制（アプリ層を信用しない）。

## 5. プロトタイプからの移行
- `profile-store.ts` → Supabaseの`student_profiles`（インターフェースは現状維持＝差し替え薄層）。`loadProfile/saveProfile`をSupabase実装に。
- `scout-store.ts` → `scouts`テーブル＋Realtime（到着即通知）。
- ダミー（students/scouts/companies）→ seedスクリプトで初期投入（サンプルは本番前に削除）。
- localStorageの既存入力は「移行ウィザード」で取り込み（任意）。

## 6. 課金（Stripe）
- **掲載プラン**：企業のSubscription（Free/Standard/Premium）。`subscriptions`。
- **スカウト課金**：従量（通数）or Standard内枠＋超過。`scout_usage`をStripe meteredへ。
- **成功報酬**：採用確定で請求書（manual invoice）。
- Webhookで`subscriptions`/支払い状態同期。`/for-companies`の料金はStripe Price連動に。
- ⚠️ **職業安定法**：有料職業紹介/募集情報等提供事業の区分・届出を**着手前に法務確認**（課金方式が区分を左右）。

## 7. AI（機能として内側）
- マッチング：`matching.ts`のヒューリスティック→ プロフィール×企業の特徴量＋LLM/embeddingで内定可能性（**参考値・断定しない**）。
- ES添削／面接／業界マッチング：Anthropic API（人間レビュー前提・YMYL）。
- **データフライホイール**：scouts/selection_recordsの結果を匿名集計→`company_stats`・マッチング学習に還流（北極星の堀）。公開する統計と、内部学習のみの非公開データを分離。

## 8. 通知
- スカウト到着/返信：Supabase Realtime＋メール（Resend）。ヘッダーの通知バッジ（`ScoutBadge`）を実データへ。

## 9. コンプラ（§7・実装ゲート）
- 同意：visibilityフラグ＝同意。RLSで強制。落ちた学生の個人公開は不可（統計のみ）。人事の声は許諾(`approved`)。
- 監査ログ（`audit_log`）、データ削除権（学生が自分のデータを削除）、利用規約/プライバシーポリシー整備。
- **法務確認（職業安定法・個人情報保護法・景表法）を本番公開前に必須**。AI判定/料金は「参考・要相談」明記を維持。

## 10. 実装フェーズ（着手はEnterPlanMode）
- **P1**：Supabase導入＋認証（LINE/Google）＋`/signup`本実装＋`student_profiles`永続化（profile-store差し替え）。
- **P2**：`scouts`＋RLS＋Realtime通知＋`/recruiter`を実データ化（企業アカウント・同意ベース表示）。
- **P3**：Stripe課金（掲載/スカウト）＋`/for-companies`連動。
- **P4**：AI実推論（マッチング/内定可能性/ES）＋データフライホイール。
- 横断：RLS/監査/規約、法務確認。

### 13.2 RLS／コンプラ（既存§4を厳格化）
- `es_entries`：本人のみ。二次利用は `share_consent=true` の**匿名・統計化**のみ（個票は企業に出さない）。
- `tokumori_scores`/`trust_scores`：本人＋（同意済みのみ）企業はバンド表示。**偏差値ゲートのスカウトはバンドで判定**（個票露出しない）。
- `member_stats_daily`：**会員数 ≥10,000 で公開ゲート開放**（M22）。それ未満は公式統計のみ。集計は夜間バッチ・実数のみ（捏造禁止）。
- `verifications`：大学メール認証（確認メール）／学生証画像（Storage＋人手or審査）。実名・認証は評価/特典/スカウト到達に加点（M23）。
- `no_shows`：本人通知・異議申立フロー必須。信頼スコア低下は説明可能に（M26）。
- **`credit_consents`（M27・最重要）**：学生信用値の外部提供は本人同意＋目的拘束。個人情報保護法／信用情報規制／割賦販売法・貸金業法等の該当性を弁護士確認。スコア根拠の説明可能性・不服申立・差別禁止を担保。

### 13.3 Stripe（§6拡張）
- 掲載＝**基本0円**（母集団形成）。課金は スカウト／**特別オファー枠**（偏差値上位への限定オファー）／成功報酬／**データプロダクト**（母集団分析・採用ベンチマーク：自社採用学生の偏差値推移 vs 業界平均）。
- 長期インターンメディアへの送客手数料・掲載。将来：信用値外部提携のレベニューシェア。

### 13.4 LINE連携（M19）
- LINE公式アカウント＋Messaging API＋webhook（サーバ必須）。会員⇔LINE友だち連携（連携コード/OAuth）。`schedule_events`をトリガに**面接の前日/当日リマインド**・締切・スカウト/特別オファー着信通知。

### 13.5 実LLM（M5/M9/M28・後フェーズ）
- ES/自己PR添削・AI面接評価・パーソナル分析を Anthropic API（Next Route Handler・`ANTHROPIC_API_KEY`サーバ専用）。当面はルールベース（`PrepAI`拡張）。YMYL＝参考・人間検証注記。

### 13.6 長期インターンメディア連携I/F（M18）
- 別メディア（別リポ想定）と**共通学生ID＋スコアAPI**で疎結合連携。受入先のインターン成果入力→`assessments`/実行力スコアへ反映（エビデンス）。掲載企業に偏差値推移を共有。

### 13.7 生きたメディア（§16・AIO堀厳守）
- 自己学習ループ（既存D7/D9・月次）＋会員行動シグナルで**トップ特集を自動入替**。適応はビルド時のみ（canonical本文不変）。Vercel Cronで計測→分析→起案→HITL→再生成。
