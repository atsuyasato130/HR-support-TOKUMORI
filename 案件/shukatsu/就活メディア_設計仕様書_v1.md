## 1. 勝ち筋と不変の設計原則

### 勝ち筋（すべての判断がここに従属）
競争優位は「AIO手法」ではない（手法は模倣可能）。堀は2つだけ：
1. **両側の一次データ（人事 × 内定者）** — 本業ネットワークでしか作れない。競合にもAIにも作れない。
2. **本業人脈による外部言及・被リンク** — 新規ドメインの立ち上げ速度の律速。

> v2の核：この堀を**自己学習ループ（§11）が自動で太らせる**。サイト内アンケートが「新しい一次データ」になり、読者が増えるほど堀が複利で太る。手法は模倣できても「自社発の一次データが自動増殖する仕組み」は競合に作れない。

### 設計の最優先順位（迷ったらこの順）
1. AIクローラーが**確実に全文取得**できる（SSG / クリーンHTML / JS非依存）
2. 一次データが**抽出されやすい構造**で提示（結論先頭・数値・短段落）
3. **エンティティ（運営者・著者・メディア）の明確さと信頼性シグナル**
4. 構造化データ（JSON-LD）は**土台として実装。過信しない**
5. （捨てる）llms.txt 等のAI特化の小細工は優先度最低
6. **【v2】適応はビルド時に行い、クローラーが見る正規HTML（canonical本文）は不変に保つ**

### 原則6の意味（D8・最重要）
「自己学習で表示が切り替わる」は**2種類に分離**する。混同するとAIO堀が崩れる。

| | 何を変えるか | クローラーへの影響 | 採否 |
|---|---|---|---|
| **ビルド時/編集時に変える** | 記事の優先順位・特集の並べ替え・テンプレ進化・新規トピック生成 → **SSG再生成** | 毎回・誰でも同じ安定HTML。**むしろ強くなる** | ✅ 主役 |
| **リクエストごとに人別で変える**（パーソナライズ） | ログイン人間向けの「次に読む」等の付加レイヤー | **正規記事本文は不変**に保てばOK | ⚠️ 付加限定 |

→ 学習結果は「リアルタイムDOM書き換え」ではなく「**次のビルドへの反映**」で実現する。気づいたら最適化されている状態を、クローラビリティを犠牲にせず作る。

### 絶対禁止事項
- **AIで一般論記事を量産しない**（AIは「一次データの展開・構造化」の道具。価値の源泉ではない）
- **人事の個別企業ホンネをそのまま公開しない**（必ず匿名化・統計化。本業クライアント信頼の破壊＝事業の死）
- **YMYL領域でAI生成の事実を人間検証なしに公開しない**

## 2. ターゲットとエンティティ階層（IA）

**ターゲット：新卒就活生・全業界横断（D2）。** サイト構造そのものをエンティティ階層とし、URLとパンくずで機械可読にする。

### 階層（主軸＝選考フェーズ / 副軸＝業界・職種）
```
就活（ルート・運営エンティティ）
├─ 選考フェーズ【主軸＝カテゴリの背骨】
│   ├─ 自己分析 / ガクチカ
│   ├─ ES（エントリーシート）
│   ├─ Webテスト・適性検査（SPI/玉手箱 等）
│   ├─ GD（グループディスカッション）
│   ├─ 面接（一次/二次/最終）
│   └─ 内定・内定承諾・オファー面談
├─ 業界・職種【副軸＝タグ/二次カテゴリ】
│   └─ コンサル / IT / メーカー / 商社 / 金融 / …（横断検索用タグ）
└─ 一次データ調査【堀＝特設】
    └─ 「現役人事N人調査」「内定者N人の実データ」「答え合わせ」シリーズ
```

### URL設計
| 種別 | URL例 | 主schema |
|------|-------|----------|
| トップ | `/` | Organization, WebSite |
| フェーズ一覧 | `/category/es` | CollectionPage, BreadcrumbList |
| 記事 | `/articles/es-gakuchika-word-count` | Article(+FAQPage) |
| 一次データ特設 | `/data/jinji-survey-2026-es` | Article, **Dataset** |
| 業界タグ | `/tag/consulting` | CollectionPage |
| 著者 | `/authors/[id]` | Person, ProfilePage |
| 運営者 | `/about` | AboutPage |

> 原則：1記事＝1検索意図＝1質問（H1）。掛け合わせ長文を作らず、薄い記事を多数で階層を太らせる方がAI可視性に有利。

## 5. 記事の型システム（AIOの核①）

### 共通骨格（全記事・上から順＝AI抽出ユニット）
1. **H1：質問形式**（例「ESのガクチカは何文字が最適？」）
2. **ConclusionBlock**：冒頭3〜5行。**確認できた事実のみ断言**（推測を断言形にしない＝誤引用事故防止）
3. **DataPoint**：数値・割合・サンプル数・調査時期（例「現役人事120人調査・2026年5月：78%が『結論から書くべき』」）
4. **本文**：各見出し直下に答え→根拠。段落2〜3文。
5. **FaqBlock**：実際にFAQがある場合のみ（表示とFAQPage schemaを一致）
6. **SourceCitation**：調査方法・サンプル数・時期・著者経歴

### 記事タイプ（content-templates として実体化）
| type | 役割 | 必須ブロック | 主schema |
|------|------|------------|----------|
| `how-to` | ノウハウ・対策 | Conclusion, 本文, FAQ任意 | Article(+FAQPage) |
| `survey` | 一次データ調査（堀） | Conclusion, **DataPoint必須**, SourceCitation必須 | Article, **Dataset** |
| `comparison` | **答え合わせ（最強の独自性）** | **AnswerMatch必須**, DataPoint, SourceCitation | Article, Dataset |
| `glossary` | 用語定義 | 「〜とは、〜である」定義文を冒頭 | Article, DefinedTerm |
| `interview` | 人事/内定者インタビュー | Conclusion, 引用, SourceCitation | Article |

### SurveyWidget の扱い（D8の徹底）
- アンケートは**付加レイヤー**として記事末尾等に置く。**正規記事本文（クローラーが見る canonical）は変えない**。
- 回答は `api/survey` で受信 → §6の一次データパイプラインへ（=新しい堀になる）。

### 文体ルール（`docs/AIO_GUIDELINES.md` に展開）
- 結論ファースト。各見出し直下に答え。段落は短く（2〜3文）。
- 「〜とは、〜である」定義文を要所に（AIが定義参照元に登録しやすい）。
- 主観形容詞を排し客観的数値・事実に置換。リスト・表で構造化。

> Phase 1成果物：`AIO_GUIDELINES.md` に「良い例／悪い例」を就活テーマで3組以上。AI生成プロンプトにこの型を埋め込む。

## 6. 一次データの設計と収集パイプライン（AIOの核②／最大の堀）

D1＝在庫ゼロのため、Phase 1の主眼は**型の確定**と**収集→記事化パイプライン設計**。

### データ2系統 ＋ 答え合わせ
- **人事側**：選考基準、ESで通る/落ちるパターン、面接で見る点、内定時期の実数、求める人物像
- **就活生側**：ES実例、選考フロー体験、内定時期、つまずき
- **答え合わせ（=記憶に残る一点）**：「学生が良いと思うES」 vs 「人事が評価するES」のギャップ → `AnswerMatch` で並置・数値化。

### 収集パイプライン（`docs/DATA_COLLECTION.md` に SOP化）
```
[1] 設問設計（人事/学生で対の質問＝答え合わせ可能に）
      ↓
[2] 取得（本業ネットワーク配布 ＋ ★サイト内SurveyWidget）
      ↓
[3] 同意取得・記録（「公開・コンテンツ利用への同意」を必須項目に・同意ログ保持）
      ↓
[4] 匿名化・統計化（個別企業名・個人特定情報を除去 → 業界全体の傾向に集約）
      ↓
[5] 人間レビューゲート（クライアント信頼を壊さないか人事担当が確認）
      ↓
[6] 構造化（survey型：サンプル数/時期/手法/設問/集計値）
      ↓
[7] 記事化（survey/comparison → AI横展開 → §10の人間レビュー → 公開）
```
> ★v2：[2]に**サイト内アンケート（SurveyWidget）**が加わる。読者が増えるほど就活生側データが自給自足で増える＝堀の複利化。

### コンプライアンス（必須・絶対）
- 取得データは**必ず匿名化・統計化**してから公開。提供者の**公開同意を取得し記録**。
- **個別企業のホンネはそのまま出さない**。必ず集約して業界傾向に。
- 個人情報・企業内部情報の扱いは本業クライアント関係に直結。

### Dataset JSON-LD（survey/comparison）
`name / description / creator(Organization) / temporalCoverage / variableMeasured / measurementTechnique / size` を機械可読に付与。

## 8. 構造化データ JSON-LD（土台・過信しない）

`lib/schema.ts` に**型付き**で実装。**画面表示を単一の真実とし自動生成**（手書き禁止＝drift防止）。

| schema | 範囲 | 要点 |
|--------|------|------|
| Organization / WebSite | layout | 運営会社名・ロゴ・URLを一貫 |
| Article | 全記事 | headline / datePublished / **dateModified（更新時必ず更新）** / author(経歴付き) / publisher |
| FAQPage | FAQ記事のみ | **画面表示のFAQと完全一致**（不一致はペナルティ要因） |
| BreadcrumbList | 全ページ | サイト階層をAIに伝える |
| Dataset | survey/comparison | 一次調査であることを機械可読に |
| Person | 著者ページ | 著者エンティティ（権威性） |

### 注意（根拠）
- JSON-LDはGEOの直接レバーではなく**土台**。2026年の複数調査でAI Overviews引用への直接効果は限定的。→ 入れるが、これで勝とうとしない。本質は§5・§6・§11。
- **マークアップと表示を必ず一致**（自動生成schemaの放置はペナルティ要因）。Microdata/RDFaは使わずJSON-LDのみ。

## 9. エンティティ・信頼性シグナル（AIOの核③・中長期で効く）

AIは「誰が言っているか」を引用の最大判断基準にする。
- **著者情報の充実**：各記事に著者明示。人事経験・業界実績など権威性を示す経歴。著者ページ（Person）に集約。
- **運営者情報の明確化**：会社概要・運営方針・連絡先。メディア名と運営会社を一貫表記（`/about`）。
- **サイト構造をエンティティ階層で**：§2の通り。
- **外部言及の獲得（本業人脈・立ち上げ速度の律速）**：プレスリリース定期配信、業界メディア言及、人事ネットワーク経由の被リンク。新規ドメインは最初引用されにくく、**ここが立ち上げ速度を決める**。Phase 1と並行で着手。

```
一次データ（§6で構造化済）
   ↓  ← AIは「一次データの横展開・構造化」だけに使う（一般論生成には使わない）
ドラフト生成（記事タイプのテンプレ＋AIO_GUIDELINESプロンプト）
   ↓
人間レビューゲート【必須】
   - 事実検証（YMYL：断言が一次データに裏付くか）
   - クライアント信頼チェック（個別企業特定の排除）
   - 著者の最終確認（署名する責任）
   ↓
microCMSへ投入 → SSG再生成 → 公開
```
- AI生成の痕跡（一般論・水増し）が残る記事は公開しない。全記事に人間著者が署名。

## 11. 自己学習ループ（自律最適化）★v2の中核

AIがAIO/SEOを計測・分析し、サイト内アンケートで一次データを自給自足しながら、**次のビルドで最適化される**閉ループ。**適応はビルド時に行い、クローラーが見る正規HTMLは不変（D8）**。

### ループ全体
```
[1] 計測(measure)  : 引用率(ChatGPT/Perplexity/Claude/AI Overview) + SEO順位 + 流入 + アンケート結果
      ↓
[2] 分析(analyze)  : どの「一次データ型 / 記事構造 / トピック」が引用・流入を生んだか
      ↓
[3] 起案(propose)  : 次に書く記事・直す構造・特集/一覧の並べ替え・テンプレ改良を AI が起案
      ↓
[4] HITLゲート      : 公開・構造変更・データ公開は人間承認（D7）
      ↓
[5] 反映           : ranking.ts更新 / microCMS更新 → SSG再生成（=表示・構成が“切り替わる”）
      ↺ [1] へ（**月次バッチ**＝Vercel Cron で月1回駆動。計測[1]のみ日次で継続蓄積）
```

### サイクル頻度（D9）
- **月次バッチ**で [2]分析→[3]起案→[5]並べ替え反映を実行（Vercel Cron 月1回）。**計測[1]は日次で継続蓄積**し、月でまとめて分析。
- なぜ月次：AIの引用シグナルは反映が遅く（新記事のクロール→索引→引用まで数週間）、週次ではノイズが大きい。初期は流入・アンケート回答も少なく、月単位でないと統計的に意味のある一次データにならない。HITL承認も月次定例の方が小チームで回る（既存の月次レポート運用と同リズム）。
- 成熟後（流入・コンテンツ量が増えたら）隔週/週次に短縮可。データが正当化したら縮める。

### 自律境界（D7・`docs/OPTIMIZATION_LOOP.md` に明文化）
| 区分 | 自動（人間承認なし） | HITL承認必須 |
|------|---------------------|--------------|
| 計測 | 引用率・SEO・流入・アンケートの収集/集計 | — |
| 分析 | 効く型/構造/トピックの分析・レポート生成 | — |
| 表示 | **特集枠・カテゴリ/記事一覧の並べ替え**（ranking.ts） | サイト構造・テンプレ・schema仕様の変更 |
| 提案 | 記事/トピックのドラフト**起案**・ダッシュボード表示 | — |
| 公開 | — | **記事の新規公開・本文/構造変更の公開・一次データ(survey/comparison)の公開** |

> 並べ替えは「同じ記事群の順序」なので正規HTMLの中身は変わらず、ビルド時確定＝クローラー安全。公開・データ公開はYMYL＋クライアント信頼に直結するので必ず人間ゲート。cc-evolutionの「設定の実変更はHITL」と同思想。

### サイト内アンケートの二重の役割
1. **ループの燃料**：どのトピックが読者に刺さるかを[2]分析に流す
2. **新しい一次データ**：回答が匿名化・統計化されて「内定者側データ」に。読者が増えるほど堀が複利で太る（§1・§6）

### 計測の取り方（実装時に手段を確定）
- **引用率**：ChatGPT/Perplexity/Claude/AI Overviewに代表質問を定期投入し、自社が引用/言及されたか記録（手動＋自動化ツール）。評価は**言及率（頻度）**。固定順位ではない。
- **SEO/流入**：Search Console / アナリティクスのAPI連携。
- **蓄積**：軽量DB（Vercel Postgres / KV 等）に時系列で保存 → analyzeが参照。
- フィードバック先：効いた一次データの型を§6の設問設計へ、効いた構造を§5テンプレ／AIO_GUIDELINESへ還流。
